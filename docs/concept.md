系统定义

一个可迭代、Vocabulary 驱动的语义编译、运行与验证系统。

它接受不完整的词汇表，通过当前语义环境解析 Prompt 和文档，生成可追溯的 Semantic IR；同时将真实使用中的用户纠错转化为语义补丁和回归用例，使基础层与领域包持续进化。

两个核心

1. 语义执行闭环

Vocabulary
→ 编译为语义包
→ 组合当前语义环境
→ 解析 Prompt / Document
→ 生成 Semantic IR
→ 澄清、执行或 Semantic Lint

2. 语义进化闭环

AI 产生错误解释
→ 用户纠正真实 Case
→ 当前实例立即修正
→ 判断错误所属层级
→ 生成有作用域的补丁候选
→ 加入回归测试
→ 用户确认或重复验证
→ 发布新的语义包版本

已经确定的产品立场

1. 这是一个系统，内部同时包含语义包构建面、运行面和迭代闭环。
2. 允许导入不完整 Vocabulary。
    只有“术语：定义”也能使用，只是能力有限。
3. Vocabulary 可以逐渐成熟：

术语与定义
→ 概念关系
→ 业务约束
→ 正例、反例和消歧规则

4. 语义展开是系统的上层能力。
    AI 可以从初级词表提出关系和约束，但推断不能静默升级为正式语义。
5. 默认只激活基础层。
    基础层是用户与 AI 的默认沟通语义，不等同于传统用户画像。
6. 领域包是显式开关。
    开启后，领域定义在其作用域内优先；没有领域定义时回退到基础层。
7. 冲突不能静默合并。
    必须保留命名空间、来源、优先级和最终绑定理由。
8. Semantic Lint 检查的是符合度。
    它判断内容是否符合当前激活的语义包，不声称判断客观真假。
9. 用户纠错是核心数据。
    每次有效纠错同时形成实例修正、纠错记录、补丁候选和回归用例。
10. 长期规则不能因一次纠错自动全局化。
    当前 Case 立即修正；长期补丁需要确认，或者经过重复证据和回归验证后晋升。
11. 解析错误与规则违反必须分开报告。
    系统需要明确“可能理解错了”与“按照当前理解确实不符合规则”的区别。
12. 系统主要迭代显式语义资产。
    包括 Vocabulary、关系、约束、示例、作用域、澄清策略和测试集，而不是默认依赖不可审计的模型参数更新。

需求基线的一句话版本

用户可以导入不完整的基础或领域 Vocabulary；系统将其编译成可组合、可版本化的语义包，并用当前激活的语义环境解析 Prompt 和文档。AI 的解释必须可追溯、可澄清、可验证。用户在真实 Case 中的纠错会被诊断、限定作用域、形成补丁和回归测试，从而持续提升未来的语义理解与 Semantic Lint 能力。

所以答案是：需求已经成功确立。

尚未确定的是 DSL 格式、Semantic IR Schema、存储结构、推理策略和模型调用方式——这些属于下一阶段的架构与实现设计，不再是产品目标不清。
